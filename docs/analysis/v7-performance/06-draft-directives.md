# 06 — Draft v7 Directives (concrete text)

The actual directive text v7 generators should emit. Three artifacts replace
today's CLAUDE.md + AGENTS.md + 19 rule files + 12 agents + 33 skill/command
descriptions. Total always-loaded budget: **≤2,500 tokens** including MCP schemas.

Design notes:

- **One statement, one place.** CLAUDE.md is the single always-loaded file for
  Claude Code. AGENTS.md (lean) serves the other tools and is *not* imported by
  CLAUDE.md — no double-loading.
- **Values, not procedures.** Each rule is one declarative line. Rationale lives
  in `.rulebook/specs/` (on demand), never in always-loaded context.
- **Orchestration is explicitly freed** (P0) — stated in the model's favor, so
  residual habits from v6-era configs don't linger.

## 6.1 Draft `CLAUDE.md` template (~55 lines, ~450 tokens)

```markdown
# {{PROJECT_NAME}}

Managed by @hivehub/rulebook v7. Few rules, all deliberate — follow all of them.

## Project
{{ONE_LINE_DESCRIPTION}}
Language: {{LANGUAGE}} · Tests: {{TEST_RUNNER}} · Docs: /docs

## Commands
- Type-check: `{{TYPECHECK_CMD}}`
- Lint: `{{LINT_CMD}}`
- Test: `{{TEST_CMD}}`
- Quality gate (run before any commit): type-check → lint → tests, all green.

## Values
1. Complete implementations — never leave stubs or TODO markers; finish, or say
   concretely why you can't.
2. Root causes, not workarounds — diagnose before changing code; type-check
   before running tests.
3. Surgical diffs — touch only what the task needs; match existing style.
4. Simplicity first — the least code that solves the problem; no unrequested
   abstractions or features.
5. Fix forward — never discard uncommitted work.
6. State assumptions — if interpretations diverge, say so instead of picking
   silently.

## Git safety (requires explicit user authorization)
`reset --hard` · `checkout -- .` / `restore .` · `clean -f` · `push --force` ·
`rebase` · `branch -D` · branch switching. Always fine: status, diff, log,
blame, add, commit.

## Orchestration
Subagents, parallel dispatch, and teams are your call — fan out freely when work
is parallel or context-heavy; work directly when it isn't. Rulebook never blocks
or mandates orchestration.

## Rulebook (on demand — no ceremony for small fixes)
- Multi-session or multi-phase work: track via the `rulebook` MCP
  (`rulebook_task`). Task checklists execute in listed order.
- Optional session context in one call: `rulebook_session start`.
- Learned something non-obvious? `rulebook_memory add`.
- Project specs: `.rulebook/specs/` — read when the work touches them.
- Analyses: `docs/analysis/<slug>/`, numbered files, one theme per file.
```

## 6.2 Draft lean `AGENTS.md` template (<3 KB, ~350 tokens)

```markdown
# Project Rules — {{PROJECT_NAME}}

Generated by @hivehub/rulebook v7 (lean). Detail lives in `.rulebook/specs/`;
read a spec when — and only when — the work touches its area.

## Rules
1. Quality gate before commit: `{{TYPECHECK_CMD}}` → `{{LINT_CMD}}` →
   `{{TEST_CMD}}`, all green. Never bypass hooks.
2. Complete implementations only — no stubs, no TODO markers, no partial work.
3. Destructive git operations require explicit user authorization
   (reset --hard, restore ., clean -f, push --force, rebase, branch -D).
4. Multi-session work is tracked as Rulebook tasks (`.rulebook/tasks/`, managed
   via the `rulebook` MCP). Checklists execute in listed order.
5. Docs in English; root allows only README/CHANGELOG/AGENTS/LICENSE/
   CONTRIBUTING/SECURITY — everything else in /docs.
6. Orchestration (subagents/parallelism/teams) is the model's choice — never
   blocked, never mandated.

## Specs index (on demand)
- `.rulebook/specs/RULEBOOK.md` — task format (SHALL/MUST + Given/When/Then)
- `.rulebook/specs/{{LANGUAGE}}.md` — language conventions
{{MODULE_SPEC_LINES}}

## Overrides
`AGENTS.override.md` — user-owned, survives `rulebook update`, wins on conflict.
```

## 6.3 Draft language rule file (the ONE `.claude/rules/` file, when detected)

Example `typescript.md` (~15 lines, ~150 tokens):

```markdown
# TypeScript

- Type-check before tests: `{{TYPECHECK_CMD}}` first, always.
- No `any` where a type is knowable; prefer `unknown` + narrowing.
- `@ts-ignore`/`@ts-expect-error` need a one-line justification.
- ESM imports with `.js` extension on relative paths.
- Interfaces for shapes, `type` for unions; `readonly` where possible.
- Use the configured test runner only; mock at module boundaries.
- Never commit dist/, build/, *.tsbuildinfo, node_modules/.
```

## 6.4 Draft `.claude/settings.json` (full-autonomy profile, F-011)

```json
{
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": [
      "Bash(*)", "Read(*)", "Edit(*)", "Write(*)",
      "Glob(*)", "Grep(*)", "Agent(*)",
      "WebFetch(*)", "WebSearch", "TodoWrite",
      "mcp__rulebook"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command", "command": "bash $CLAUDE_PROJECT_DIR/.claude/hooks/protect-task-scaffolding.sh" }]
      }
    ]
  }
}
```

No permission prompts for anything the model routinely does; the single
path-only guard protects task scaffolding; safety lives in directive values
(git safety) and the quality gate, not in interruptions. Rulebook only adds
rules — user-authored permissions and enterprise policies always win.
