# 07 — `/analysis <topic>` Command + Skill

**User directive (2026-04-07):** every time the user asks to "analyze X", the same repeatable process must run — exactly the workflow that produced *this very document*. No more ad-hoc analyses; every investigation accumulates into the knowledge base so future implementations consult prior findings first.

## Workflow steps

1. **Slugify** the topic → `<slug>` (e.g. `v5.3.0`, `auth-rewrite`, `perf-startup`)
2. **Create directory** `docs/analysis/<slug>/` with skeleton files:
   - `README.md` — entry point (executive summary, links to themed files)
   - `findings.md` — numbered findings list (F-001, F-002, …)
   - `execution-plan.md` — phased plan that becomes tasks
   - Optional supporting files per sub-area (one per theme — exactly like this v5.3.0 directory)
3. **Agent selection** — based on topic keywords, pick relevant subagents in parallel:
   - Code-heavy → `Explore` + `feature-dev:code-explorer`
   - Architecture → `architect` + `feature-dev:code-architect`
   - Security → `security-reviewer`
   - Performance → `performance-engineer`
   - Always include `researcher` as baseline
   - User can override via `/analysis <topic> --agents <list>`
4. **Parallel investigation** — agents run concurrently, each writes a section file in `docs/analysis/<slug>/`
5. **Consolidation** — main thread merges agent outputs into `README.md` with:
   - Executive summary (top findings)
   - **Numbered findings** F-001 … F-NNN, each with: title, evidence (file:line), impact, confidence
   - Cross-references between findings
6. **Execution plan generation** → `execution-plan.md` with phases and per-phase task list
7. **Task materialization** — for each phase, automatically call `rulebook task create` producing real tasks under `.rulebook/tasks/`. Each generated task:
   - Links back to the analysis: `Source: docs/analysis/<slug>/README.md#F-NNN`
   - Carries the mandatory docs+tests+verify tail (see [06-task-tail.md](./06-task-tail.md))
   - Is scoped to ≤2 files where possible (per `task-decomposition.md` rule)
8. **Knowledge base capture** — every numbered finding is registered:
   - `rulebook_knowledge_add` for patterns/anti-patterns discovered
   - `rulebook_learn_capture` for implementation insights
   - `rulebook_decision_create` for architectural choices that emerged
   - All entries tagged `analysis:<slug>` so they're queryable as a cohort
9. **Memory write** — `rulebook_memory_save` with `type: analysis`, full README content, tags `[analysis, <slug>, <detected-topics>]`
10. **Pre-implementation gate** — when any task generated by an analysis is later started, the task tooling automatically surfaces the source analysis and runs `rulebook_knowledge_search` + `rulebook_memory_search` filtered by `analysis:<slug>`. The model is **required** to consult these before writing code (enforced by a new rule `templates/rules/consult-analysis-before-implementing.md`).

## Behavioral guarantees

- **Idempotent**: re-running `/analysis <existing-slug>` updates findings (preserves IDs, appends new ones) instead of overwriting
- **Auditable**: each finding has a `discovered-by:` agent attribution and `confidence:` score
- **Replayable**: `analysis-manager` writes a `manifest.json` listing agents, prompts, model versions, timestamps
- **Linked**: `rulebook task list` shows `[analysis:<slug>]` badge on tasks generated by an analysis
- **Themed splitting (NEW)**: when a consolidated `README.md` exceeds ~400 lines, the workflow auto-splits content into themed `NN-<theme>.md` files and reduces `README.md` to a thin index — exactly the structure of this v5.3.0 directory.

## Why this is P0

The current `docs/analysis/v5.3.0/` directory is a one-off proof that the workflow produces high-value output. Codifying it means every future "analyze X" request compounds the knowledge base instead of producing throwaway notes. It also closes a gap in the existing `knowledge-base-usage.md` rule, which says "consult before implementing" but provides no structured way to *populate* the KB at the right granularity — `/analysis` is that producer.

## Open question
