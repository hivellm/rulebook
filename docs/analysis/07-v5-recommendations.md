# Rulebook v5: Concrete Recommendations

Based on analysis of UzEngine (game engine, 43 agents) and TML (programming language, 32 agents).

---

## Vision

Rulebook v5 should be a **tool-agnostic AI development framework**. Today Rulebook generates templates. Tomorrow it should generate **complete workflows** that work with Claude Code, Cursor, Gemini CLI, Codex, Windsurf, Continue.dev — or any future tool.

**Core principle**: Quality directives are universal. Delivery is tool-specific.

---

## Multi-Tool Architecture: 3 Layers

### Layer 1: Universal Directives (any LLM reads these)

Markdown files in the root or `.rulebook/specs/` that **any tool** can read:

| File | Purpose | Read by |
|------|---------|--------|
| `AGENTS.md` | Index + Tier 1 rules | Codex, Copilot, any LLM |
| `.rulebook/specs/RULEBOOK.md` | Task management | Any tool with file access |
| `.rulebook/specs/QUALITY_ENFORCEMENT.md` | Quality gates | Any tool |
| `.rulebook/specs/TIER1_PROHIBITIONS.md` | **NEW** — Absolute prohibitions | Any tool |
| `.rulebook/specs/<LANGUAGE>.md` | Language-specific rules | Any tool |
| `.rulebook/PLANS.md` | Session scratchpad | Any tool |

**These directives are written as plain text** — they don't depend on agents, MCP, or tool-specific features. A model reading `TIER1_PROHIBITIONS.md` knows exactly what's forbidden, whether it's Claude, GPT, Gemini, or Llama.

### Layer 2: Tool-Specific Configuration (generated by Rulebook)

Rulebook detects which tools are in use and generates configs for each:

```
rulebook init
# Detected: Claude Code + Cursor
# Generating:
#   AGENTS.md (universal)
#   CLAUDE.md (Claude Code)
#   .claude/agents/*.md (Claude Code agents)
#   .claude/rules/*.md (modular rules)
#   .cursor/rules/*.mdc (Cursor rules)
#   .cursor/mcp.json (MCP for Cursor)
```

**Generation map by tool:**

| Feature | Claude Code | Cursor | Gemini CLI | Codex | Windsurf | Continue.dev | Copilot |
|---------|-------------|--------|------------|-------|----------|-------------|---------|
| **Directives file** | `CLAUDE.md` | `.cursor/rules/*.mdc` | `GEMINI.md` | `AGENTS.md` | `.windsurf/rules/*.md` | `.continue/rules/*.md` | `.github/copilot-instructions.md` |
| **Specialized agents** | `.claude/agents/*.md` | N/A (contextual rules) | N/A (inline) | N/A (inline) | N/A (inline) | N/A (inline) | N/A |
| **Modular rules** | `.claude/rules/*.md` | `.cursor/rules/*.mdc` | Sections in GEMINI.md | Sections in AGENTS.md | `.windsurf/rules/*.md` | `.continue/rules/*.md` | Sections in instructions.md |
| **Persistent memory** | `.claude/agent-memory/` | Via MCP | Via MCP | Via MCP | Via MCP | Via MCP | N/A |
| **Multi-agent** | Teams + SendMessage | N/A | N/A | N/A | N/A | N/A | N/A |
| **MCP** | Native | `.cursor/mcp.json` | `mcp.json` | Partial | `mcp.json` | `mcp.json` | N/A |
| **Quality gates** | Pre-commit hooks | Pre-commit hooks | Pre-commit hooks | Pre-commit hooks | Pre-commit hooks | Pre-commit hooks | Pre-commit hooks |

### Layer 3: MCP (cross-tool, works with any MCP-enabled tool)

The Rulebook MCP server is already cross-tool. Expand to cover features that individual tools don't support natively:

| MCP Tool | Purpose | Replaces native feature from |
|----------|---------|------------------------------|
| `rulebook_memory_save/search` | Persistent memory | agent-memory (Claude Code only) |
| `rulebook_task_*` | Task management | No tool has this natively |
| `rulebook_decision_*` | ADRs | No tool has this |
| `rulebook_learn_*` | Learnings | No tool has this |
| `rulebook_knowledge_*` | Knowledge base | No tool has this |
| **NEW** `rulebook_rules_list` | List active rules | `.claude/rules/` (Claude/Cursor only) |
| **NEW** `rulebook_session_start` | Load session context | Manual PLANS.md reading |
| **NEW** `rulebook_session_end` | Save session summary | Manual PLANS.md writing |
| **NEW** `rulebook_blockers` | Visualize blocker chain | No tool has this |

## R1: Tiered Rule System (Universal)

### What to build
Replace flat rule generation with a 3-tier system, generated as **universal markdown** + **tool-specific configs**:

```
Tier 1: ABSOLUTE PROHIBITIONS (generated for EVERY tool)
  - No stubs, TODOs, placeholders, approximations, partial implementations
  - No rm/delete without authorization
  - No destructive git operations
  - No guessing — research first
  - Sequential file editing

Tier 2: MANDATORY WORKFLOW (configurable per project)
  - Task decomposition threshold (default: 3 files)
  - Specialist delegation (when available in the tool)
  - Data flow planning (for cross-subsystem changes)
  - Incremental test development

Tier 3: STANDARDS (auto-detected from project)
  - Language conventions, naming, formatting
  - Framework-specific patterns
  - Build/test commands
```

### Generation by tool

**TIER1_PROHIBITIONS.md** (universal) — generated once, read by all:
```markdown
# Absolute Prohibitions (Tier 1)

## PROHIBITION 1: No Shortcuts
NEVER simplify logic, add TODO/FIXME/HACK, create stubs, use placeholders,
reduce scope, skip edge cases, or deliver partial implementations.
Quality is everything. Response time is irrelevant.
...
```

**Cursor** (`.cursor/rules/tier1-prohibitions.mdc`):
```yaml
---
description: "Absolute prohibitions — ALWAYS apply"
alwaysApply: true
---
[same content as TIER1_PROHIBITIONS.md]
```

**Gemini** (section in `GEMINI.md`):
```markdown
## HIGHEST PRECEDENCE RULES
[same content]
```

**Copilot** (section in `.github/copilot-instructions.md`):
```markdown
## Mandatory Rules
[same content]
```

The **same directive**, adapted to each tool's format.

## R2: Adaptive Agent Framework

### Problem
Only Claude Code supports custom agents (`.claude/agents/`). Cursor, Gemini, Codex have no equivalent.

### Solution: Graceful Degradation

**For Claude Code** (full support):
```
.claude/agents/
├── shader-engineer.md       (full specialized agent)
├── cpp-core-expert.md       (with pre-flight checklist)
├── systems-integration.md   (planning agent)
└── researcher.md            (read-only, haiku)
```

**For Cursor** (contextual rules that simulate specialization):
```
.cursor/rules/
├── shader-engineering.mdc     (applied when editing .hlsl/.glsl)
├── cpp-core.mdc              (applied when editing .cpp/.h)
├── quality-checklist.mdc     (pre-flight checklist, always active)
└── task-management.mdc       (task rules, always active)
```

Cursor doesn't have agents, but it has **contextual rules with frontmatter** that activate based on file type or description. Rulebook can generate rules that **simulate** specialization.

**For Gemini/Codex/Windsurf** (inline in directives file):
```markdown
# GEMINI.md

## When Editing Shaders (.hlsl, .glsl)
[shader-engineer rules injected here]

## When Editing C++ (.cpp, .h)
[cpp-core-expert rules injected here]

## Pre-Delivery Checklist (MANDATORY)
[pre-flight checklist injected here]
```

### Configuration
```json
{
  "agentFramework": {
    "projectType": "game-engine",
    "tools": ["claude-code", "cursor"],
    "agents": {
      "shader-engineer": {
        "domain": "shaders",
        "filePatterns": ["*.hlsl", "*.glsl", "*.msl", "*.wgsl"],
        "tier": "core",
        "checklist": ["source-citation", "no-approximations", "all-parameters"]
      },
      "cpp-core-expert": {
        "domain": "cpp",
        "filePatterns": ["*.cpp", "*.h", "*.hpp"],
        "tier": "core",
        "checklist": ["raii", "const-correctness", "engine-apis"]
      }
    }
  }
}
```

The generator produces:
- Claude Code: full agent definition with memory
- Cursor: .mdc rule with `globs` in frontmatter
- Gemini/Codex: conditional section in directives file

### Agent Template Library (by project type)

**Game Engine**:
- shader-engineer (core), cpp-core-expert (core), rhi-engineer (core)
- render-graph-engineer (core), ecs-systems-engineer (core)

**Web Application (Full-Stack)**:

**Mobile Application**:

## R3: Rule Modularization (Cross-Tool)

### What to build
Rules live in `.rulebook/rules/` (canonical) and are **projected** to each tool:

```
.rulebook/rules/                     <- canonical source
├── no-shortcuts.md                  (Tier 1)
├── git-safety.md                    (Tier 1)
├── no-delete.md                     (Tier 1)
├── sequential-editing.md            (Tier 1)
├── task-decomposition.md            (Tier 2)
├── research-first.md                (Tier 2)
├── incremental-tests.md             (Tier 2)
├── no-deferred.md                   (Tier 2)
└── <project-specific>.md            (user-created)

# Auto-generated by `rulebook update`:
.claude/rules/                       <- projection for Claude Code
.cursor/rules/                       <- projection for Cursor (.mdc)
```

### Canonical rule format
```markdown
---
name: no-shortcuts
tier: 1
description: "Never use stubs, TODOs, placeholders, or approximations"
alwaysApply: true
filePatterns: ["*"]     # applies to all files
tools: ["all"]          # generate for all tools
---

# No Shortcuts — Quality Over Speed

## Absolute Prohibitions
1. NEVER simplify logic to deliver faster — implement the correct algorithm
2. NEVER add TODOs — if it needs to be done, do it NOW
3. NEVER use stubs — every function must have its real implementation
...
```

The `rulebook update` command reads `.rulebook/rules/` and generates:
- `.claude/rules/no-shortcuts.md` (Claude Code format)
- `.cursor/rules/no-shortcuts.mdc` (Cursor format with YAML frontmatter)
- Section in `GEMINI.md` (inline format)
- Section in `.github/copilot-instructions.md`
- Section in `.windsurf/rules/no-shortcuts.md`

**One source, multiple projections.**

## R4: Reference Implementation Workflow (Universal)

### Configuration
```json
{
  "referenceSource": {
    "enabled": true,
    "path": "W:\\UE_5.6_Source",
    "type": "source-tree",
    "citationFormat": "@source UE5 <file>:<line>",
    "mandatoryFor": ["rendering", "physics"],
    "comparisonTool": {
      "type": "ir-comparison",
      "referenceCompiler": "rustc --emit=llvm-ir",
      "projectCompiler": "tml build --emit-ir"
    }
  }
}
```

### Universal generation
Regardless of tool, the directive is the same:
```markdown
## MANDATORY: Consult Reference Before Implementing

Before writing any code for [rendering, physics]:
1. Find the equivalent implementation in [W:\UE_5.6_Source]
2. Read the reference source code completely
3. Only then implement with citation: [@source UE5 <file>:<line>]
4. If you cannot find the reference -> STOP and research first
```

Works with any LLM. The difference: in Claude Code there's a dedicated `researcher` agent; in Cursor/Gemini, the assistant does the research itself.

### Pre-flight checklist generation
When reference source is configured, generate domain-specific checklists:
```markdown
## Pre-Flight Checklist (MANDATORY)
1. Which reference source file was this based on? (path:line REQUIRED)
2. List every numeric constant and its reference source
3. Are ALL parameters from the reference present? (must be YES)
4. Are ALL defaults identical? (must be YES)
5. Is any runtime computation replaced by hardcoded values? (must be NO)
```

## R5: Cross-Tool Persistent Memory

### Problem
Only Claude Code has `agent-memory/`. Cursor, Gemini, Codex lose context between sessions.

### Solution: MCP as universal memory layer

```
Any tool with MCP:
  rulebook_memory_save -> save context
  rulebook_memory_search -> search previous context
  rulebook_session_start -> load PLANS.md + relevant memories
  rulebook_session_end -> save session summary

Tools without MCP:
  .rulebook/PLANS.md -> manual scratchpad read/write
  .rulebook/memory/ -> SQLite database queryable via CLI
  `rulebook memory search "query"` -> terminal search
```

### Structured PLANS.md (universal)
```markdown
<!-- PLANS:CONTEXT:START -->
## Current Focus
[What the team is working on right now]

## Key Decisions
[Important decisions made this session]

## Blockers
[Active blockers and their status]
<!-- PLANS:CONTEXT:END -->

<!-- PLANS:TASK:START -->
## Active Task
[Current task details, progress, sub-tasks]
<!-- PLANS:TASK:END -->

<!-- PLANS:HISTORY:START -->
## Session Log
### 2026-03-19
- Completed: [summary]
- Discovered: [key findings]
- Next: [what to pick up next session]
<!-- PLANS:HISTORY:END -->
```

**Works with any tool** — it's just a markdown file the LLM reads at session start.

## R6: Enhanced Task Management

### Task dependencies
```json
{
  "blocks": ["task-2", "task-3"],
  "blockedBy": ["task-0"],
  "cascadeImpact": 5
}
```

### Anti-deferred enforcement
- "Deferred" status is forbidden
- Alternative: "blocked" with explicit blocker reference
- Directive: implement the blocker first, or explain why it's truly impossible

### Blocker chain (via CLI or MCP)
```bash
rulebook task blockers
# B1: async-runtime -> blocks: task-5, task-6, task-7 (3 tasks, HIGH impact)
# B2: lambda-conversion -> blocks: task-8, task-9 (2 tasks, MEDIUM impact)
```

## R7: Token Optimization Templates (Universal)

Verbosity rules by model tier (tool-agnostic labels):

**Small models / limited context** (research tier: Haiku, Gemini Flash-Lite, GPT-4o-mini):
```markdown
## Output Rules
- Output code, not explanations
- Minimal reports: "Done" instead of detailed status
- No markdown tables, emoji, or status sections
- Combine outputs into one response
```

**Mid-tier models** (standard tier: Sonnet, Gemini Flash, GPT-4o):
```markdown
## Output Rules
- Brief summaries (2-3 sentences max)
- Code with inline comments (no separate explanation blocks)
- Report only: what changed, what passed, what failed
```

**Large models** (core tier: Opus, Gemini Pro, o3):
```markdown
## Output Rules
- Full explanations welcome for complex decisions
- Document reasoning for non-obvious choices
- Detailed analysis for bug investigations
```

## R8: Multi-Agent with Graceful Degradation

| Feature | Claude Code | Cursor | Gemini/Codex/Windsurf |
|---------|------------|--------|----------------------|
| **Parallel agents** | Teams + SendMessage | N/A -> contextual rules | N/A -> sequential workflow |
| **File ownership** | Agent-specific | N/A -> manual rule | N/A -> manual rule |
| **Research + implement** | researcher(haiku) -> implementer(opus) | Single assistant does both | Single assistant does both |
| **Coordination** | team-lead orchestrates | User orchestrates | User orchestrates |

### For tools without multi-agent support
Generate an **equivalent sequential workflow**:
```markdown
## Workflow for Multi-Subsystem Tasks

Since this tool doesn't support parallel agents, follow this sequence:

### Phase 1: Research
- Read ALL relevant files BEFORE editing any
- Create a change plan in `docs/analysis/plan-<feature>.md`
- Map dependencies between files

### Phase 2: Implementation (one file at a time)
- Edit files in dependency order (upstream first)
- Build/test after each file
- NEVER edit 3+ files without the Phase 1 plan

### Phase 3: Verification
- Run complete quality gates
- Verify end-to-end integration
- Update tasks.md
```

## Success Metrics

Rulebook v5 is successful when:

1. **Zero manual rule files needed** — Projects like UzEngine and TML don't need 17+ custom files
2. **Works with any tool** — Same quality directives for Claude Code, Cursor, Gemini, Codex
3. **One source, multiple projections** — `.rulebook/rules/` generates configs for all detected tools
4. **3x faster setup** — From `rulebook init` to productive AI development in minutes, not days
5. **Graceful degradation** — Tools without agents receive equivalent sequential workflows
6. **Cross-tool memory** — MCP + PLANS.md preserve context regardless of tool
7. **Cost reduction** — Tier assignment reduces API costs by 40-60% without quality loss
